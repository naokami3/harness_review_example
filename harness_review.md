## 指摘一覧

### 指摘1: JWT署名検証が無効化されている [Critical]

**確認方法**: コンテナ内でRubyを使い、正規と異なるシークレット（`'totally-wrong-secret'`）で署名したJWTを生成し、`GET /tasks`にリクエストした。

**確認結果**: HTTP 200が返り、タスク一覧が正常に取得できた。署名検証が行われていないため、任意のシークレットで署名したトークンが通る。

**該当コード**: `app/controllers/application_controller.rb:8`
```ruby
decoded = JWT.decode(token, secret_key, false)
```
第3引数`false`が署名検証を無効化している。

**影響**: 攻撃者は任意の`user_id`を含むJWTを偽造でき、全ユーザーになりすましが可能。

---

### 指摘2: 期限切れJWTが認証を通過する [Critical]

**確認方法**: コンテナ内でRubyを使い、`exp`を1時間前に設定した正規署名のJWTを生成し、`GET /tasks`にリクエストした。

**確認結果**: HTTP 200が返り、認証が通った。期限切れトークンで全操作が可能。

**原因**: 指摘1と同じく`JWT.decode`の第3引数が`false`のため、`exp`クレームの検証もスキップされている。

---

### 指摘3: 他ユーザーのタスクを取得・更新・削除できる（IDOR） [Critical]

**確認方法**: ユーザーA（id=4）でタスク（id=30）を作成後、ユーザーB（id=5）のトークンで以下を実行した：
- `GET /tasks/30` → 200、ユーザーAのタスク内容が全て返った
- `PATCH /tasks/30` → 200、タイトルが「HACKED BY USER B」に書き換わった
- `DELETE /tasks/30` → 200、`deleted_at`がセットされた

**確認結果**: 3操作全て成功。認証済みであれば他ユーザーのリソースを自由に操作できる。

**該当コード**: `app/controllers/tasks_controller.rb:47-49`
```ruby
def set_task
  @task = Task.find(params[:id])
end
```
`current_user`によるスコープがないため、タスクIDを知っていれば誰のタスクでも操作可能。

---

### 指摘4: 論理削除したタスクが一覧・フィルタに表示される [High]

**確認方法**: タスクを作成 → `DELETE`で論理削除 → `GET /tasks`で一覧取得。

**確認結果**: 削除済みタスク（`deleted_at`が非NULL）が一覧に含まれた。全27件中2件が削除済みのまま返された。`GET /tasks/by_status/todo`でも同様に削除済みタスクが含まれた。

**該当コード**: `app/controllers/tasks_controller.rb:10`
```ruby
tasks = Task.where(user: current_user).offset(offset).limit(per_page)
```
`where(deleted_at: nil)`が抜けている。`by_status`アクション（41行目）も同様。

---

### 指摘5: ページネーションが1ページ目を読み飛ばす [High]

**確認方法**: ユーザーAで27件のタスクを作成し、以下を試した：
- `GET /tasks`（page未指定）→ 20件返った（offset=0、正常）
- `GET /tasks?page=1` → 7件返った（offset=20、最初の20件がスキップされた）
- `GET /tasks?page=2` → 0件

**確認結果**: `page=1`で1ページ目を取得するとoffset=20になり、最初の20件が取得できない。

**該当コード**: `app/controllers/tasks_controller.rb:6-8`
```ruby
page     = params[:page].to_i
per_page = (params[:per_page] || 20).to_i
offset   = page * per_page
```
正しくは`offset = (page - 1) * per_page`。現在の実装ではpage=0が実質的な1ページ目になっている。

---

### 指摘6: priorityとstatusにバリデーションがない [Medium]

**確認方法**: 以下のリクエストを実行した：
- `POST /tasks` with `priority: -1` → 201で保存された
- `POST /tasks` with `priority: 999999` → 201で保存された
- `POST /tasks` with `status: "invalid"` → 201で保存された

**確認結果**: 任意の値が保存できる。priorityに負数や極端な値、statusに不正な文字列をそのまま格納可能。

**該当コード**: `app/models/task.rb`
```ruby
class Task < ApplicationRecord
  belongs_to :user
  validates :title, presence: true
end
```
`title`の存在チェックのみで、`priority`と`status`のバリデーションがない。

---

### 指摘7: 存在しないリソースへのアクセスでスタックトレースが露出する [Medium]

**確認方法**: `GET /tasks/99999`を認証済みでリクエストした。

**確認結果**: HTTP 404が返ったが、レスポンスボディにRailsのフルスタックトレース（ファイルパス・gem内部の行番号含む）が含まれた。development環境の`consider_all_requests_local = true`によるもの。本番環境では`rescue_from ActiveRecord::RecordNotFound`でJSONエラーレスポンスを返すべき。

---

### 指摘8: JWTシークレットキーにハードコードされたフォールバックがある [Medium]

**確認方法**: コードリーディングで確認。`application_controller.rb:19`に`'fallback-secret'`がハードコードされている。

**確認結果**: `JWT_SECRET_KEY`環境変数が未設定の場合、推測可能なシークレットが使用される。docker-compose.ymlでは設定されているが、設定漏れ時のリスクが残る。

## 単独レビュー（solo_review.md）との差分

### 単独レビューで指摘されていて、本レビューでも確認できた項目

| 単独レビューの指摘 | 本レビューでの確認結果 |
|---|---|
| JWT署名未検証（Critical） | 実際に偽署名トークンで認証を突破できた |
| IDOR脆弱性（Critical） | 実際に他ユーザーのタスクを更新・削除できた |
| 論理削除の不備（High） | 実際に削除済みタスクが一覧に含まれた |
| ページネーション0始まり（Medium） | page=1で1ページ目が取得できないことを確認した |
| status/priorityバリデーション不足（Low） | 実際にpriority=-1, status="invalid"が保存できた |
| シークレットキーのフォールバック（High→Medium） | コードで確認（動作影響は環境変数設定時にはない） |

### 本レビュー（ハーネス）で新たに確認できた項目

| 項目 | 内容 |
|---|---|
| 期限切れJWTの認証突破 | 単独レビューでは「署名未検証」として一括りだったが、実際に期限切れトークンで認証が通ることを個別に確認した |
| by_statusでも論理削除が反映されない | 単独レビューでは`index`のみ言及だったが、`by_status`でも同じ問題を確認した |
| ページネーションの具体的な挙動 | 単独レビューでは「0始まりで混乱する」という指摘だったが、実際にはpage=1で最初の20件が完全にスキップされる致命的な動作であることを確認した。重要度をMedium→Highに引き上げた |
| 存在しないリソースでスタックトレースが露出 | 単独レビューでは指摘されていなかった |
| テストが実行環境によって不安定 | requestスペックが初回403で失敗し、再実行で通った。テスト基盤の信頼性に問題がある |

### ハーネスレビューの優位性

単独レビュー（コードリーディングのみ）は全ての主要バグを指摘できていたが、以下の差がある：

1. **重要度の精度**: ページネーションの問題を単独レビューは「0始まりで混乱する」（Medium）としたが、実際に動かすと「1ページ目が完全に消失する」（High）だった。コードを読んだだけでは影響度を過小評価した
2. **漏れの発見**: `by_status`エンドポイントの論理削除漏れ、スタックトレース露出は単独レビューでは指摘されていなかった
3. **確証の有無**: 単独レビューの指摘は全て「〜のリスクがある」という推測形だが、本レビューでは全て「実際に〜できた」という事実として報告できた
